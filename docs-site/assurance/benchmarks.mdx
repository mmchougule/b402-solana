Per [`PRD-A05`](https://github.com/b402-ai/b402-solana-assurance/blob/main/prds/PRD-A05-benchmarks-budgets.md),
the assurance harness measures off-chain wall times and on-chain
compute units / tx sizes. These are regression-tracked: a CU spike on
shield from 240k to 400k is a regression even if it still works.

## Off-chain wall times

From [`reports/benchmark-summary.md`](https://github.com/b402-ai/b402-solana-assurance/blob/main/reports/benchmark-summary.md):

```
Platform:                       darwin arm64
Node:                           v24.4.0

Wallet build avg ms:            0.740
Merkle append avg ms:           3.797
Note encrypt + decrypt avg ms:  3.064
Proof generation avg ms:        733.357   ← dominates
Verifier CLI avg ms:            116.791
Adapt fixture load avg ms:      0.093
Note scan + decrypt avg ms:     3.086
```

Proof generation is **97% of the off-chain wall time** for any
shielded action. Optimization paths:

- **Rapidsnark (C++).** Drop-in Groth16 prover; ~3× faster than
  snarkjs WASM. v1 stays on snarkjs for portability and zero-install
  browser compatibility. v2 should land Rapidsnark for server-side
  prover hosts.
- **Recursion.** PRD-17 explores Halo2-style recursive proofs to fold
  multiple notes into one. Out-of-scope for v1.
- **Worker threads.** snarkjs proofs are single-threaded JS. Wrap in
  a worker for non-blocking client UX. Trivial.

## On-chain compute units

From [`artifacts/onchain-compute-units.json`](https://github.com/b402-ai/b402-solana-assurance/blob/main/artifacts/onchain-compute-units.json):

| Scenario | LiteSVM CU | Devnet pin |
|---|---|---|
| `shield_success` | 233,495 | 239,224 |
| `unshield_success` | 223,413 | 350,000 |
| `adapt_execute_full_path` (real Groth16 verifier-adapt CPI, mock adapter) | 313,725 | — |

`devnet pin` is the value that the LiteSVM harness asserts the devnet
reading does not exceed; sourced from the README's observed-cost table.

Mainnet-fork measurements from the live `swap-e2e-jupiter` run on the
same `b402-pool` bytecode:

| Path | Tx size | CU |
|---|---|---|
| Shield (devnet) | 1,157 B | 239,224 |
| Unshield (devnet) | ~1,150 B | ~350,000 |
| Adapt swap, mock adapter | 1,214 B | ~600,000 |
| Adapt swap, real Jupiter v6 SolFi V2 | 1,231 B | ~660,000 |

The LiteSVM `adapt_execute_full_path` (313,725 CU) is the pool's
own work plus one adapter CPI minus the actual downstream
Jupiter+AMM cost. The devnet number for the same shape (mock adapter,
real Groth16) is ~600k CU; the delta is the pool's pinned CU-budget
request and the test-mock invariant overhead. The Jupiter mainnet-fork
line at ~660k adds the real aggregator + SolFi V2 pool CPIs.

Pool-side overhead is ~325k CU (313,725 LiteSVM probe + minor
budget-request difference). This leaves ~1.07M CU of headroom under
Solana's 1.4M cap — the remaining budget is what an adapter spends on
the downstream protocol.

## CU budget per instruction

From [`PRD-03 §8`](https://github.com/b402-ai/b402-solana/blob/main/docs/prds/PRD-03-program-spec.md#8-compute-budget-reservation),
SDK helpers set these CU limits:

| Instruction | Requested CU (PRD-03 §8) | Observed CU |
|---|---|---|
| init_pool | 300k | — |
| add_token_config | 100k | — |
| shield | 350k | 233,495 (LiteSVM), 239,224 (devnet) |
| transact | 400k | — |
| unshield | 400k | 223,413 (LiteSVM), ~350k (devnet) |
| adapt_execute (mock adapter, full path) | 900k | 313,725 (LiteSVM), ~600k (devnet) |
| adapt_execute (Jupiter 2-hop, mainnet-fork) | 900k | ~660k (real Jupiter SolFi V2) |
| adapt_execute (Kamino) | 700k | — (mainnet-fork deposit verified, CU not pinned in artifacts) |
| adapt_execute (Drift) | 1,100k | — (deferred) |
| admin ops | 200k | — |

Headroom is on purpose — leaving ~30% slack accounts for protocol
upgrades that increase costs.

## Tx-size budgets

Solana caps a v0 transaction at **1,232 bytes serialized**. b402's
observed sizes:

| Op | Size | Headroom |
|---|---|---|
| Shield | 1,157 B | 75 B |
| Unshield | ~1,150 B | ~80 B |
| Adapt (mock) | 1,214 B | 18 B |
| Adapt (Jupiter mainnet-fork) | 1,231 B | 1 B |

The Jupiter case is **at the edge**. This is why the b402 ALT
(`9FPYufa1KDkrn1VgfjkR7R667hbnTA7CNtmy38QcsuNj` on devnet) is
non-optional for adapt: without it, Jupiter routes longer than 2 hops
overflow the cap, and even 2-hop can overflow depending on which AMMs
the route uses.

`MAX_ACTION_PAYLOAD = 350` (set by the SDK) leaves enough headroom
for 23 public inputs × 32 B = 736 B + 256 B proof + Borsh overhead +
~80 B of accounts + dummy nullifiers + encrypted notes.

## CU regression policy

The assurance harness tracks per-instruction CU as a baseline. A 10%
deviation from baseline triggers a CI flag. The numbers in this page
are the current baseline.

Specific regressions to watch:

- **Tree append CU.** Each `sol_poseidon` call is ~100k CU. If a
  hot path regresses to >2 Poseidon calls per merkle level, we're
  missing an optimization.
- **Verifier CU.** Should stay near 178k. Drift indicates `groth16-solana`
  changed in a way we should review.
- **Adapter CPI overhead.** The pool-side overhead (excluding the
  adapter's own work + its downstream CPI) should stay near 200k.
