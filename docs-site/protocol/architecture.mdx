## Instruction taxonomy

The pool program exposes these instructions (full spec in
[Pool Program](/protocol/pool-program)):

```
init_pool                 admin, one-time
add_token_config          admin, whitelist mint
shield                    user, deposit
transact                  user, internal transfer
unshield                  user, withdraw to clear
adapt_execute             user, composable DeFi
register_adapter          admin
disable_adapter           admin (single-sig kill switch)
enable_adapter            admin (multisig)
pause / unpause           admin
queue_upgrade             admin (72h timelock)
activate_upgrade          admin (post-timelock)
revoke_upgrade_authority  admin (irreversible after 12mo)
rotate_admin              admin
```

## State (PDAs)

All PDAs use `b"b402/v1"` as a version prefix in seeds. Layout from
[`PRD-03 §3.3`](https://github.com/b402-ai/b402-solana/blob/main/docs/prds/PRD-03-program-spec.md):

| PDA | Seeds | Purpose |
|---|---|---|
| `PoolConfig` | `[b"b402/v1", b"config"]` | Admin pubkey, pause flags, pinned verifier program IDs |
| `TokenConfig` | `[b"b402/v1", b"token", mint]` | Per-token: decimals, vault pubkey, enabled |
| `Vault` (token account) | `[b"b402/v1", b"vault", mint]` | Pool's custody of `mint` |
| `TreeState` | `[b"b402/v1", b"tree"]` | 128-root ring, frontier, leaf count, zero cache |
| `NullifierShard` | `[b"b402/v1", b"null", prefix_le_2B]` | Nullifier set, sharded by 16-bit prefix |
| `AdapterRegistry` | `[b"b402/v1", b"adapters"]` | Registered adapter program IDs + allowed instructions |
| `TreasuryConfig` | `[b"b402/v1", b"treasury"]` | Reserved for future fee policy (currently unused) |

`NullifierShard` is created lazily on first nullifier landing in shard.
Up to 65,536 shards in saturation; ~10 KB each, ~64 KB at expected
load.

## Cross-program invocation graph

For a typical `adapt_execute` call:

```
relayer keypair signs the tx
   │
   ▼
b402_pool::adapt_execute
   ├─ CPI → b402_verifier_adapt::verify (~180k CU)
   ├─ CPI → spl-token::transfer  (in_vault → adapter_in_ta, pool PDA signs)
   ├─ CPI → adapter_program::execute
   │       ├─ CPI → spl-token::transfer (adapter scratch)
   │       ├─ CPI → jupiter_v6::route   (adapter PDA signs)
   │       │       └─ CPI → AMM programs (Whirlpool, SolFi, Raydium, ...)
   │       └─ CPI → spl-token::transfer (adapter_out_ta → out_vault)
   ├─ post-CPI: read out_vault.amount
   ├─ CPI → spl-token::transfer (in_vault → relayer_fee_ta if fee > 0)
   ├─ tree_append (sol_poseidon × 26 levels)
   └─ emit AdaptExecuted, CommitmentAppended events
```

Compute budget: 1.4M CU requested, ~660k actual on the mainnet-fork
Jupiter case ([README §observed costs](https://github.com/b402-ai/b402-solana/blob/main/README.md)).

## Construction class vs EVM privacy pools

b402-solana uses the same Groth16-on-BN254 + Poseidon + UTXO + nullifier
construction that Railgun, Aztec, and Tornado Cash all use. That
construction class is shared; the implementation here is independent —
Circom 2 sources, Anchor programs, SVM-native CPI semantics, no shared
code with any EVM project.

Differences vs the canonical EVM design (using Railgun as the most
similar reference):

| | EVM Railgun | b402-solana |
|---|---|---|
| Composition primitive | `delegatecall` to RelayAdapt contract | `adapt_execute` instruction → CPI to per-protocol adapter program |
| Fee payer | EntryPoint (ERC-4337) | Solana fee-payer keypair (distinct from signers; no AA layer needed) |
| Pool state | Multiple slots in one contract | Multiple PDAs (TreeState, NullifierShard, Vault per mint) |
| Verifier | One contract per circuit | One program per circuit, immutable (`upgrade_authority = None`) |
| Tree append cost | Block gas | One Solana instruction; full adapt-execute observed at ~660k CU |
| Tx-size cap | ~30k bytes (block-dependent) | 1,232 B hard cap, ALT-relieved |
| Account budget | Unlimited | ~35 accounts per tx without ALT, ~256 with one ALT |

The runtime difference matters: Solana's read-write set is explicit per
transaction, so the b402 tree append fits in a single instruction —
the runtime knows `TreeState` is locked for this tx and serializes
appends without a changelog. EVM privacy pools rely on implicit slot
locking and pay for it in gas variability.

## What's intentionally stubbed (per README)

From the README's "What's intentionally stubbed":

- **Stealth-address bech32 encoding.** Primitives exist; the
  `b402sol1q…` encode/decode layer is not written.
- **Production trusted setup.** Current VK is from a throwaway
  single-contributor ceremony.
- **Admin multisig.** v1 uses single-key admin; PRD-03 §9 specifies
  the full co-signer pattern.
- **Beyond Jupiter + Kamino-deposit.** Kamino withdraw/borrow/repay,
  Drift, Adrena, Orca are scoped per
  [PRD-09 / PRD-10 / PRD-16](https://github.com/b402-ai/b402-solana/tree/main/docs/prds)
  but not yet shipped.

The relayer service was originally listed as a stub; it has since
shipped (`@b402ai/solana-relayer`) — see
[Running a Relayer](/operations/running-a-relayer).
