## Localnet

`./ops/local-validator.sh --reset` boots `solana-test-validator` with
all 4 programs deployed at deterministic IDs. The IDs are constructed
from the keypairs in `ops/keypairs/` (gitignored).

For repeatable testing across machines, the localnet IDs differ per
operator; reference them via `Anchor.toml` rather than hardcoding.

## Verifying programs match the source

To confirm a deployed devnet program matches a known source commit:

```bash
# 1. Build the program for BPF.
cargo build-sbf --tools-version v1.54 \
  --manifest-path programs/b402-pool/Cargo.toml

# 2. Hash the resulting .so.
sha256sum target/deploy/b402_pool.so

# 3. Pull the on-chain program data.
solana -u devnet program dump 42a3hsCXtQLWonyxWZosaaCJCweYYKMrvNd25p1Jrt2y \
  /tmp/onchain.so
sha256sum /tmp/onchain.so

# Hashes must match modulo Anchor's metadata trailer.
```

For an extra-paranoid verification, use
`cargo-build-bpf --verifiable` (when available) for fully reproducible
builds.

## Migration log

When the pool's program ID changes (which happens at every fresh
deployment), the migration policy from
[`PRD-01 §4.4`](https://github.com/b402-ai/b402-solana/blob/main/docs/prds/PRD-01-architecture.md#44-migration-policy)
applies:

1. Old pool stays operational indefinitely so users can always exit.
2. New pool deploys with new program ID + new verifier.
3. SDK exposes a `migrationUnshield` flow that proves ownership in the
   old pool and shields into the new pool atomically.
4. Users are never force-migrated; migration is opt-in.

For tracking deployments:

- Devnet IDs are pinned in `Anchor.toml` per deployment.
- Mainnet IDs will be published in `ops/deploy/mainnet-ids.toml` after
  deployment with full deploy artifacts (slot, deployer pubkey,
  program-data hash).
